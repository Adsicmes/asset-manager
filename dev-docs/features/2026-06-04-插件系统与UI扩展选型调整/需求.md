# 插件系统与 UI 扩展选型调整需求

> 创建日期: 2026-06-04 10:33
> 最后更新: 2026-06-04 10:33
> 作者: Adsicmes
> 状态: 草稿

## 背景

上一版技术选型已确定项目优先使用 Windows 原生桌面技术栈。当前需要进一步强化插件系统，使插件不仅能扩展后台处理逻辑，还能扩展文件类型预览、素材处理动作、独立操作面板和主界面元素。插件编写门槛需要尽量低，避免每个插件都要理解复杂的 Windows Shell、WPF 内部结构或宿主私有实现。

## 目标

调整技术栈和插件架构，使插件作者可以用简单的 C# + Razor 方式扩展素材能力和 UI。宿主需要提供稳定的插件 SDK、UI 插槽、文件能力授权、预览协议和操作面板协议。

## 用户故事

作为插件作者，我希望通过少量 C# 代码声明支持的文件扩展名、预览组件和处理动作，以便快速扩展新的素材格式。

作为插件作者，我希望可以给素材详情、预览区、工具栏、右键菜单和设置页增加自己的 UI，以便把插件功能融入主程序工作流。

作为用户，我希望安装插件后，新文件类型可以自动获得预览、元数据提取、批处理和专属操作面板，以便不必等待主程序更新。

## 功能描述

### 核心功能

- [ ] 插件可声明支持的文件扩展名、MIME 类型、文件签名和预览优先级。
- [ ] 插件可提供预览渲染器、元数据提取器、缩略图生成器、批处理动作和导出动作。
- [ ] 插件可注册独立操作面板，例如文件格式专属工具、批处理控制台、导出设置页。
- [ ] 插件可向宿主 UI 插槽注册按钮、菜单项、侧栏分区、详情页分区、搜索过滤器、状态栏元素和设置页。
- [ ] 插件 SDK 需要提供项目模板、属性标注、默认基类、宿主服务接口和打包命令。

### 边界条件

- 插件不得直接持有主窗口、WPF VisualTree 或内部 ViewModel 的可变引用。
- UI 扩展通过宿主注册的 slot 完成，不允许插件绕过宿主布局和权限系统任意修改界面。
- 不可信插件不得加载到主进程；其 UI 和处理逻辑需要走进程外宿主或 WebView2 沙箱。
- 插件访问文件必须通过宿主能力授权，不能默认获得全磁盘读写权限。

## 非功能需求

- 易用性: 一个最小插件应能通过模板创建，并在 20 行左右代码内声明文件类型、预览和操作。
- 稳定性: 插件崩溃不能导致主程序崩溃；插件 UI 渲染失败应只影响对应面板。
- 可维护性: 插件 SDK 的 public API 必须版本化，宿主内部重构不应破坏已发布插件。
- 一致性: 插件贡献的简单按钮、菜单、标签、状态元素由宿主用原生 WPF 样式渲染，保持整体 UI 一致。

## 验收标准

1. 插件可以新增一个未知扩展名的文件预览能力。
2. 插件可以为选中素材增加右键动作和批处理动作。
3. 插件可以在详情侧栏和预览区加入自己的操作面板。
4. 插件作者可以通过 `dotnet new` 模板创建插件项目并打包。
5. 不可信插件不能直接访问宿主内部对象或未经授权的文件路径。

## 优先级与排期

- 优先级: P0
- 预计工期: 待实现规划确认
- 依赖项: WPF + Blazor Hybrid 原型、插件 SDK 原型、UI slot registry 原型、预览协议原型
